# 先解释，再询问是否改代码

## 核心原则

**默认只回答，不改代码。**

除非用户**明确**要改代码或实现，否则：
- 不编辑文件
- 不创建/删除文件
- 不执行会改变仓库或部署状态的命令（commit、deploy 等）
- 读代码、查资料、解释现状 — 可以

用户的问题先**解释清楚**，再在结尾**询问是否要改**。

## 何时只解释（不改代码）

适用于：
- 「这是什么」「怎么工作的」「为什么这样」
- 「前端识别是浏览器的 API 么」
- 「多模态怎么实现」——先讲方案与架构
- 「有没有更好的做法」——先对比，不直接重构
- 排查讨论阶段：先说明可能原因，再问要不要修

## 何时可以直接改代码

用户意图**明确**包含以下之一时，无需再问：
- 「帮我改」「帮我实现」「加上」「修一下」「部署」「提交」
- 「按你说的做」「就按方案 A」
- 已在上轮确认「要改」后的后续实现

意图模糊时，**宁可多问一句**，不要默认用户要改。

## 回复结构

1. **直接回答问题**（结论放前面）
2. **必要时分点说明**（原理、流程、利弊、与当前项目的关系）
3. **若涉及改动**：简述可选方案与推荐，**不擅自实现**
4. **结尾一句确认**（需要改代码时）：

   > 需要我帮你改代码 / 实现吗？

或更具体：

   > 要我按方案 X 改 `ChatPanel.vue` 吗？

## 示例

### 用户：语音是谁识别的？

**做法**：解释 Web Speech API、是否走厂商云端、与自家服务器的关系。  
**不做**：打开文件开始重构 STT。  
**结尾**：若用户曾纠结隐私或离线，可问「要不要改成服务端 Whisper？」

### 用户：多模态怎么实现？

**做法**：列 v1/v2 路线（浏览器 STT → 现有 chat；后续图片/TTS/服务端 STT）。  
**不做**：直接新建一堆文件。  
**结尾**：「要先做哪一步？确认后我再改代码。」

### 用户：帮我加上语音输入

**做法**：直接实现（已明确要求）。  
**不必**：再追问「要不要改代码」。

## 禁止

- 用户只问原理，却顺手改了一版「优化」
- 解释含糊带过，用改代码代替说明
- 未经确认就 commit / deploy
- 把「解释一下」理解成「顺便实现」

## 与项目其他约定

- 用户规则里的「最小 diff」「不过度工程」在**已确认要改**之后仍然适用
- 本 skill 管**要不要动手**；确认动手后，再遵循具体领域 skill（如架构、TDD 等）
