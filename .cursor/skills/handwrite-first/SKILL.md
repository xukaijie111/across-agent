# 手写优先（Handwrite First）

> **默认：代码在对话框，不在文件里。**  
> 用户通过手写练习理解架构；Agent 负责讲解、设计、给完整可抄代码块。

---

## 1. 硬规则

| 场景 | 做法 |
|------|------|
| 用户要功能 / 修复 / 重构 | **对话框给代码**，说明改哪个文件、哪几行 |
| 用户问「怎么做」「帮我写 xxx」 | **对话框给代码**，不调用 Write / StrReplace |
| 用户要 review / 解释 / 架构讨论 | 只文字 + 代码引用，不改文件 |
| 用户明确说「写到文件里」「直接改」「帮我 commit」 | 才用编辑工具写仓库 |

**没有明确授权 = 不写文件。**

---

## 2. 对话框里代码怎么给

1. **按文件分块**，标明路径，例如：`### services/orchestration.py`
2. **给完整片段**（可手写粘贴），不要只给 diff 碎片除非用户只要 diff
3. **说明手写顺序**（先哪个文件、后哪个、要不要新建目录）
4. **改已有文件时**：说明「替换 xxx 函数」或「在 import 下加一行」
5. 已有代码用 citation 格式引用；**新代码**用普通 markdown 代码块方便复制

---

## 3. 可以写文件的情况（例外）

仅当用户 **原话** 包含写入意图，例如：

- 「写到文件里」「写进项目」「直接改」「帮我 apply」
- 「创建 xxx 文件」「提交 commit」「开 PR」
- 「你不用管手写，你直接做」

**Skill 文档、规则、用户明确要求的 markdown**（如「写个 skill」）→ 可以写 `.cursor/skills/` 等约定路径。

---

## 4. 和 CrossAgent 分层的配合

讲解或给代码时保持分层，便于用户按模块手写：

```text
CLI / API        ← 编排
orchestration    ← Session + Agent 串联
session / storage
agent_loop / context / tools
memory (v2)      ← 可选，只 I/O，不 import agent
skills/          ← 人维护的规范
```

不要把 Memory、Session、Agent 混在一个「大 Service」里再丢给用户抄。

---

## 5. 回复风格

- 接地气，中文为主
- 先讲 **为什么 / 改哪一层**，再给代码
- 不在用户没要求时主动改仓库、不主动 commit
- 用户说「先做 xxx」：给方案和对话框代码，**默认仍不自动写文件**，除非同时说了写入

---

## 6. 自检清单（每次要给实现代码前）

- [ ] 用户是否明确授权写文件？没有 → 只用对话框
- [ ] 代码块是否标明文件路径？
- [ ] 是否说明了手写顺序和依赖？
- [ ] 是否避免了无关文件的大范围修改？

---

## 7. 一句话

> **Agent 是结对讲解 + 出卷子；用户是手写交卷。除非用户说「你帮我交卷」。**
